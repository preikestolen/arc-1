## TL;DR

The "new MCP protocol" is the **2026-07-28** release (currently RC, RC-locked 2026-05-21,
final 2026-07-28). Its headline is **statelessness**: the `initialize` handshake and
`Mcp-Session-Id` session are removed; serverâ†’client requests (elicitation/sampling) become
**multi-round-trip** instead of a held-open SSE stream; async **Tasks** graduate to an
opt-in extension; OAuth is hardened; **MCP Apps** (server-rendered UI) is added.

**For a typical MCP server this is a big, breaking migration. For ARC-1 it is small-to-medium
and mostly mechanical** â€” because ARC-1 already made the two bets the new spec forces on
everyone else:

1. ARC-1 already runs **stateless** MCP (`sessionIdGenerator: undefined`, fresh `Server` +
   transport per request â€” [src/server/http.ts:159](../../src/server/http.ts)). The change that
   terrifies stateful servers is the model ARC-1 already ships.
2. The **SAP ADT lock is fully contained inside one tool call** and never crosses an MCP
   boundary. The "lock mechanism issue" the ecosystem worries about **does not apply to ARC-1's
   current design** (details in Â§4).

Real work is: **SDK v1 â†’ v2 upgrade** (package split + import churn, ~8 files; trigger-gated,
see ADR-0006), a **CORS header addition** (shipped 2026-07-20), and an **OAuth-hardening audit**
(mostly already done). Nothing in the SAP/ADT layer is touched.

**Is the lock affected? No** â€” not by the current architecture. It only becomes a concern if
ARC-1 later adopts async Tasks for *writes*; Â§5 records the invariant that prevents that.

### SDK reality

The **TypeScript SDK v2 is in beta** and speaks 2026-07-28 end-to-end (server answers
`server/discover`; client probes it and falls back to `initialize` for old servers). It is a
**major restructure**: the monolithic `@modelcontextprotocol/sdk` splits into
`@modelcontextprotocol/server`, `/client`, and thin framework adapters (Node/Express/Hono/
Fastify), and now ships **both ESM and CJS**. v2 passes the conformance suite except the tasks
suite (deferred). **Stable v2 lands 2026-07-28.** ARC-1 is on `@modelcontextprotocol/sdk`
`^1.28.0` (resolved 1.29.0).

## 2. ARC-1's MCP surface (what a bump touches)

Confirmed by reading the code â€” the wire-facing footprint is tiny and concentrated:

- **Server:** low-level `Server`, capabilities `{ tools: {} }` only â€” no resources/prompts/

  `structuredContent` anywhere.

## 4. The lock mechanism question â€” answered

**The ecosystem concern is real:** a stateless, async-first protocol is hostile to servers that
**hold a lock/stateful resource across the request lifecycle**. If elicitation returns to the
client mid-operation (MRTR) or work is deferred to a poll-later Task, any lock held in process
memory across that gap either leaks or lands on a different (stateless) server instance that
doesn't hold it. SAP ADT's `lock â†’ modify â†’ unlock` is exactly the kind of stateful resource
that would be at risk **if it spanned an MCP round-trip**.

**It does not, in ARC-1.** Verified:

- Every writer runs `lock â†’ modify â†’ unlock` inside a single `withStatefulSession` block, with a
  guaranteed `finally` unlock ([src/adt/crud.ts:219](../../src/adt/crud.ts) and every handler in
  `src/handlers/write/`). The `lockHandle` is a local variable â€” **no module-level lock map, no
  lock cached across calls.**
- `withStatefulSession` spins up a **throwaway** ADT HTTP client whose stateful cookies live only
  for that callback ([src/adt/http.ts:266](../../src/adt/http.ts)). The
  `X-sap-adt-sessiontype: stateful` header is an **SAP-HTTP** concept, fully orthogonal to MCP
  session state.
- ARC-1 **never elicits while holding a lock** â€” the write path contains no `elicitInput`, and
  the elicit helpers are production-dead, test-only code (Â§2). So MRTR (SEP-2322) has nothing to interrupt.
- Because MCP is already stateless in ARC-1, there is no MCP session that could even in principle
  outlive a lock.

**Conclusion:** an MCP spec bump **cannot change ARC-1 lock semantics** â€” no lock state crosses a
tool-call boundary today. The lock is safe precisely because it is synchronous and self-contained.

## 5. The one invariant to add

The lock is safe *by current design*, not by rule. The single forward-looking risk is a future
feature that breaks the containment. Record this as an architectural invariant (candidate for
AGENTS.md "Security & Architectural Invariants" and/or an ADR):

> **ADT locks never cross an MCP round-trip.** A write's `lock â†’ modify â†’ unlock` must complete
> inside one synchronous tool call (one `withStatefulSession` block). Do **not** move elicitation,
> sampling, or any serverâ†’client request inside the lock block, and do **not** expose writes as
> async **Tasks** that hold the lock across the poll window â€” under the stateless protocol a later
> poll can land on a different instance that does not hold the lock, and a held ADT lock blocks
> other users. If a long write ever needs Tasks, the Task must *not* own the lock: acquire/release
> the lock within each synchronous sub-step, or make the operation idempotent and lock-free.

Any read-only long operation (e.g. a big ATC run, mass where-used) *is* a fair candidate for Tasks
later â€” reads hold no lock.
