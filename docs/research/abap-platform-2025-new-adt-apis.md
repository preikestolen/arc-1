# New ADT APIs on ABAP Platform 2025 (SAP_BASIS 816) ‚Äî Deep Dive

**Status:** research / evidence-gathering. Not implemented.
**Date:** 2026-06-04.
**Systems:** `a4h-2025.marianzeis.de:50100` (ABAP Platform 2025, SAP_BASIS **816**, S/4HANA 2025) vs `a4h.marianzeis.de:50000` (S/4HANA 2023, SAP_BASIS **758**). Both probed live as user `MARIAN`, client 001.
**Companion artifacts:** probe fixture `tests/fixtures/probe/abap-platform-2025-onprem-trial/`; memory `project_sap_2025_816_system`. Builds on [PR #347](https://github.com/marianfoo/arc-1/pull/347), which added 816 as a validated probe target.

---

## 1. Why this document exists

The 816 system's ADT discovery document (`/sap/bc/adt/discovery`) advertises **352 collections** vs **232** on the 2023 system ‚Äî **124 new collection URLs**. Collapsing the boilerplate sub-resources (`$configuration`, `$schema`, `source/formatter`) leaves **44 distinct new endpoints**. This doc establishes, for each: what it is, how it behaves on the wire (live-tested), whether it is genuinely 2025-only, whether it is useful to ARC-1, and how we would implement support.

### Methodology (how every claim here was verified)

1. **Discovery diff.** Parsed `discoveryMapKeys` from the committed 2025 and 2023 probe fixtures ‚Üí the 124/44 deltas.
2. **Live smoke test.** `GET` each of the 44 base endpoints on **both** systems with the Accept type from the 2025 discovery doc. 2023 ‚Üí `404` confirms "genuinely new"; the 2025 status + content-type reveals the call shape.
3. **Object-type decoding.** Cross-referenced every 4-letter code against SAP's own [`SAP/abap-file-formats`](https://github.com/SAP/abap-file-formats) repo (the canonical source of object-type code + description + JSON file-format schema) and the ADT-for-Eclipse 3.52/3.54 release notes (the clients paired with S/4HANA 2025 FPS00).
4. **Schema/format confirmation.** Fetched live `$schema` and a working CDS-test-cases response to prove the generic mechanism.

## 2. Headline findings

1. **39 of the 44 new endpoints are genuinely 2025-only** (2023 returns `404 ExceptionResourceNotFound`). The other 5 (`navigation/target`, `crosstrace/traces`, `packages/settings`, `packages/$constraints`, `aps/iam/suso/...`) already exist on 2023 (returned `405`/`406`, not `404`) ‚Äî only their discovery registration/shape changed; treat them as **not new**.

2. **The new endpoints split cleanly into two classes:**
   - **Class A ‚Äî "server-driven" repository objects** (‚âà30 endpoints). Bare `GET` ‚Üí **`400`** ("needs object name/params"); read/write a single object via `‚Ä¶/{name}` and `‚Ä¶/{name}/source/main` with content-type **`application/vnd.sap.adt.blues.v1+xml`**. These are *developer repository objects* with a transportable file format. **One mechanism covers them all.**
   - **Class B ‚Äî tooling / AI / governance action endpoints** (‚âà9 endpoints). Bare `GET` ‚Üí **`405`** (POST-only) or a query-driven `GET`. Not repository objects ‚Äî they're operations (code prediction, chat, explain, CDS test-double generation, ATC exemptions).

3. **The unifying mechanism for Class A is the AFF / "server-driven object types" framework.** Every Class-A object exposes `‚Ä¶/$schema` returning a **JSON Schema (draft 2020-12) auto-generated from `SAP/abap-file-formats`**, under `application/vnd.sap.adt.serverdriven.schema.v1+json; framework=objectTypes.v1`. Live proof (DESD):
   ```
   GET /sap/bc/adt/ddic/desd/$schema
   Accept: application/vnd.sap.adt.serverdriven.schema.v1+json; framework=objectTypes.v1
   ‚Üí 200; body: {"$comment":"This file is autogenerated ‚Ä¶ see https://github.com/SAP/abap-file-formats ‚Ä¶",
                 "$id":"‚Ä¶/file-formats/desd/desd-v1.json","title":"CDS Logical External Schema", ‚Ä¶}
   ```
   **Implication for ARC-1:** a *single* generic read/write path ‚Äî "read the server's schema, GET/PUT `blues.v1+xml` source by name" ‚Äî would unlock the entire Class-A set at once, instead of per-type plumbing.

## 4. Class A ‚Äî server-driven repository objects (the big opportunity)

### 4.1 How they work on the wire (verified)

All Class-A objects share the AFF generic-object contract. Using DTSC as the worked example:

```
# Discovery advertises:
collection href=/sap/bc/adt/ddic/dtsc/sources  title="Static Cache"
  accept: application/vnd.sap.adt.blues.v1+xml, text/html
  templateLink: /sap/bc/adt/ddic/dtsc/sources/{object_name}{?corrNr,lockHandle,version,accessMode,_action}
  templateLink: /sap/bc/adt/ddic/dtsc/sources/{object_name}/source/main{?corrNr,lockHandle,version}

# Read object metadata:   GET ‚Ä¶/dtsc/sources/{name}          Accept: application/vnd.sap.adt.blues.v1+xml
# Read object source:     GET ‚Ä¶/dtsc/sources/{name}/source/main
# Object schema (format): GET ‚Ä¶/ddic/desd/$schema            Accept: application/vnd.sap.adt.serverdriven.schema.v1+json; framework=objectTypes.v1
```

Verified routing (read-by-name reaches the handler; the object simply doesn't exist on the trial):
```
GET /sap/bc/adt/ddic/dtsc/sources/ZNOPE_DTSC  ‚Üí 404 "Error while reading the object ZNOPE_DTSC from the database"
GET /sap/bc/adt/conn/commtargets/ZNOPE_CT     ‚Üí 404 "Error while importing object ZNOPE_CT from the database"
```
(404 = *resource not found*, not 404 = *endpoint not registered*; the latter is what 2023 returns for the whole collection.)

> **Note:** the trial ships no custom instances of these new types, so we could not capture a `200` source read of a real object. The read-by-name **routing** is confirmed; an end-to-end `200` read/round-trip needs a seeded object (follow-up).

### 4.3 How we'd implement Class A in ARC-1 (one generic path)

ARC-1 today reads/writes AFF objects (DDLS, BDEF, SRVD, SRVB, ‚Ä¶) through **per-type** code in `src/handlers/intent.ts` (`objectBasePath`, `buildCreateXml`) + `src/adt/client.ts`. Class A is large and homogeneous, so the per-type approach scales poorly. The 816 framework hands us a better path:

**Recommended: a generic "server-driven object" (SDO) read/write path.**
- **Read** (`SAPRead type=<CODE>`): resolve the collection URL from the discovery map (ARC-1 already caches it ‚Äî `src/adt/discovery.ts`), then `GET ‚Ä¶/{name}` (`blues.v1+xml`) for metadata and `GET ‚Ä¶/{name}/source/main` for source. No per-type URL table needed ‚Äî discovery already maps `category ‚Üí href`.
- **Write** (`SAPWrite`): lock ‚Üí `PUT ‚Ä¶/{name}/source/main` (`blues.v1+xml`) ‚Üí unlock ‚Üí activate, using the **server-provided `$schema`** (`‚Ä¶/$schema`) to validate the payload pre-write (mirrors today's AFF validator in `src/aff/`, but with the schema fetched live instead of bundled).
- **Discovery-gate** each type (like the existing transport-target gate): only expose `type=DTSC|DESD|CSNM|‚Ä¶` when the collection is present in the discovery map, so 2023/7.50 systems cleanly report "unavailable" instead of 404-ing.
- **Files touched** (sketch): `src/adt/client.ts` (a `getServerDrivenObject(category,name)`), `src/handlers/intent.ts` (`SAPRead`/`SAPWrite` dispatch for the new codes via discovery lookup), `src/handlers/{schemas,tools}.ts` (enum the new type codes), `src/aff/validator.ts` (accept a live-fetched schema), `src/probe/catalog.ts` (add the new types with `minRelease: 816`).

**Why this is attractive:** one implementation + tests covers ~15 object types, all gated by discovery, all sharing one content type and one schema mechanism. It also retro-improves how ARC-1 could handle *future* AFF object types without code changes.

**Caveat:** `blues.v1+xml`/`serverdriven.schema.v1+json` are the AFF "generic object handling" content types; ARC-1 has not used them before. A spike should round-trip **one** real object (seed a DTSC or a Communication Target on the trial, read‚Üíedit‚Üíwrite‚Üíactivate) before committing to the generic path. Some types may have type-specific create prerequisites (e.g. DTSC requires an existing CDS table entity).

## 5. Class B ‚Äî tooling / AI / governance endpoints

These are **operations, not objects** ‚Äî none should be modeled as a repository type. Usefulness varies:

### 5.1 CDS Test-Double Framework ‚Äî `aunit/dbtestdoubles/cds/testcases` ‚≠ê (most useful to ARC-1)
The one Class-B endpoint that is directly, immediately useful. Live-verified working:
```
GET /sap/bc/adt/aunit/dbtestdoubles/cds/testcases?ddlsourceName=I_LANGUAGE
Accept: application/vnd.sap.adt.aunit.dbtestdoubles.cds.testcases.v1+xml
‚Üí 200:
  <cdstestcases:root><cdstestcases:cds>I_LANGUAGE</cdstestcases:cds>
    <cdstestcases:testCases><cdstestcase>
      <title>Test I_LANGUAGE.</title><testMethod>test_cds_view</testMethod>
      <description>Test CDS View as a whole.</description><semanticType>NONE</semanticType>
    </cdstestcase></cdstestcases:testCases></cdstestcases:root>
```
It scaffolds CDS unit-test cases (test method + semantics) for any CDS view. **ARC-1 value: HIGH** ‚Äî a natural `SAPDiagnose action=cds_testcases` (or `SAPLint`/test helper) that returns a ready test skeleton for a CDS entity. Sibling `‚Ä¶/testdata` and `‚Ä¶/testmethod` (POST) build out the data/method bodies. Implementation: thin `client.getCdsTestCases(ddls)` + a read-scoped action; gated on the endpoint's discovery presence (816+).

### 5.2 ATC exemptions ‚Äî `atc/remarks` (POST)
Create/approve/reject ATC exemption requests + reviewer remarks, transportable. **ARC-1 value: Medium** ‚Äî complements `SAPDiagnose action=atc`; an operator could request an exemption for a false-positive. Write-scoped, governance-sensitive (needs approver authorization `SAP_A4C_BC_DEV_CHKE_APR_PC`). Defer unless ATC workflows are in scope.

### 5.3 AI endpoints ‚Äî `$codeprediction`, `chat/conversations`, `abapunit/explain`, `abapunit/ai/chat/action`
SAP's **Joule for Developers** (licensed, SAP Note 3571857): predictive completion, conversational chat, AI explain, AI unit-test assist. **ARC-1 value: NONE to consume** ‚Äî these are SAP's own AI layer, parallel/competitive to ARC-1's value proposition (an LLM already drives ARC-1). Track for competitive awareness (see `reference_sapse_adt_vscode_mcp` memory); do not integrate.

## 6. Recommendations for ARC-1

**Priorities (highest first):**
1. **CDS test-case scaffolding** (`aunit/dbtestdoubles/cds/testcases`) ‚Äî small, self-contained, immediately useful, read-scoped. Best first follow-up.
2. **Generic server-driven object (SDO) read path** ‚Äî spike a single round-trip on a real seeded object (DTSC or Communication Target), then add a discovery-gated generic reader covering ~15 new types at once. Read before write.
3. **EVTO/EVTB (RAP business events)** ‚Äî if/when ARC-1 deepens RAP support, these slot into the SDO path and complement BDEF/SRVD/SRVB.
4. **ATC exemptions** ‚Äî only alongside broader ATC-workflow features.

**Explicitly NOT recommended:** the AI endpoints (`$codeprediction`, `chat`, `explain`) ‚Äî competitive, not consumable.

**Cross-cutting:** every new type must be **discovery-gated** (present in `src/adt/discovery.ts`'s cached map) so ARC-1 degrades cleanly on 758/7.50, exactly like the existing transport-target capability gate. Add each with `minRelease: 816` to `src/probe/catalog.ts` so the probe tracks availability across systems.
